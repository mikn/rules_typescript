```bash
# Build the bundle (concatenates all .js files in placeholder mode, or uses Vite if configured)
bazel build //src/app:app_bundle

# Run the bundled output (requires Node.js)
node bazel-bin/src/app/app_bundle.es.js
```

**Difference from rules_go — NO native "run" target:**

Unlike `go_binary`, which creates an executable that can be directly invoked, `ts_binary` produces a JavaScript bundle that requires a Node.js runtime. There is **no direct `bazel run //src/app:app_bundle`** equivalent in the base rules.

To make a runnable target, you'd need to create a custom wrapper:

```starlark
sh_binary(
    name = "app",
    srcs = [":run_app.sh"],
    data = [":app_bundle"],
)

# run_app.sh:
# #!/bin/bash
# node "$(dirname "$0")/app_bundle.es.js"
```

Then `bazel run //src/app:app` would work.

---

### --- Difference ---

| Aspect | rules_go | rules_typescript |
|----------|----------|-----------------|
| **Binary type** | Static executable (ELF, Mach-O, PE) | JavaScript bundle (.js) + Node.js runtime |
| **Direct `bazel run` support** | YES — native go_binary | NO — ts_binary produces .js only |
| **Output format** | Single machine-code executable | ESM / CJS / IIFE bundle |
| **External dependencies** | Statically linked | Requires Node.js at runtime |
| **Bundler** | Implicit (Go linker) | Explicit (Vite or custom, via ts_bundle/ts_binary) |

**Friction point:** TypeScript developers expect bundling to be automatic, but rules_typescript requires explicit configuration of `ts_binary` with a bundler. The `ts_binary` rule is named to parallel `go_binary`, but the semantics are quite different.

**Behavior:**


| Aspect | rules_go | rules_typescript |
|--------|----------|-----------------|
| **Type checking** | Mandatory, always runs | Optional (configured in .bazelrc) |
| **Failure behavior** | Type errors block the build | Type errors fail validation output group |
| **Skip type-checking** | Not possible | `--output_groups=-_validation` |
| **Tool** | Native Go compiler | tsgo (Go port of TypeScript) |
| **Speed** | Fast (type info cached) | Slower (tsgo is a full type-checker, validates entire graph) |

**Key insight — Validation actions:**

TypeScript type-checking runs as a **Bazel validation action** in the `_validation` output group. This is unusual compared to rules_go:

- `bazel build //lib` builds .js + .d.ts files
- `bazel build //lib --output_groups=+_validation` also runs tsgo type-checking
- Type errors do NOT block downstream actions (unlike Go)
- But by default, `--output_groups=+_validation` in .bazelrc makes type errors fail the entire build

**Friction point:** The validation action abstraction is non-obvious to newcomers. Developers may be confused why type errors sometimes fail the build and sometimes don't (depending on `--output_groups`).

**Note: ts_binary is the current stable API, ts_bundle is the implementation. Both have identical attrs.**

**Setup with Vite bundler:**

**BUILD.bazel (root):**
```starlark
load("@rules_typescript//ts:defs.bzl", "ts_binary")
load("@rules_typescript//vite:bundler.bzl", "vite_bundler")

ts_binary(
    name = "app_bundle",
    entry_point = "//src/app",
    bundler = ":vite",          # Optional: use Vite for real bundling
    format = "esm",             # "esm", "cjs", or "iife"
    sourcemap = True,           # Generate .js.map files

        "API_URL": "https://api.example.com",
        "DEBUG": "false",
    },
)
```

**Advanced options:**

```starlark
ts_binary(
    name = "app_advanced",
    entry_point = "//src/app",
    bundler = ":vite",
    format = "esm",
    sourcemap = True,
    
    # Chunk splitting (lib mode only)
    split_chunks = True,        # Produces a directory with multiple chunks
    
    # Code splitting for specific packages
    external = [
        "react",                # Don't bundle React
        "react-dom",            # Loaded separately at runtime
    ],

    },

```

```starlark
load("@rules_typescript//ts:defs.bzl", "ts_compile")

ts_compile(
    name = "my_lib",  # <-- Overridden by directive
    srcs = ["index.ts"],

```

**Example gazelle_ts.json in src/ directory:**

```json
{
  "pathAliases": {
    "@/": "src/",
    "@components/": "src/components/",
    "@utils/": "src/utils/"
  },
  "excludePatterns": ["*.snap", ".stories.tsx"],
  "excludeDirs": [".next", "dist"],
  "runtimeDeps": {
    "test": ["@npm//:happy-dom"]
  }
}
```

**Framework detection:**

Gazelle auto-detects frameworks from `package.json`:

```json
{
  "dependencies": {
    "@tanstack/react-router": "1.x",  // <-- Detected as TanStack framework
    "@tanstack/start": "1.x"
  }
}
```

When detected, Gazelle may generate different rules or exclusions (currently a no-op for TanStack, but prepared for future extensions).

### --- Difference ---

| Aspect | rules_go | rules_typescript |
|--------|----------|-----------------|
| **Directives** | `gazelle:prefix`, `gazelle:ignore`, `gazelle:resolve`, etc. | `gazelle:ts_package_boundary`, `gazelle:ts_ignore`, `gazelle:ts_target_name`, `gazelle:ts_warn_unresolved` |
| **Configuration file** | None (directives only) | `gazelle_ts.json` (optional, per-directory) |
| **Path alias support** | Implicit (Go imports are absolute) | Explicit via `gazelle_ts.json` |
| **Framework detection** | Implicit (Go's module system) | Explicit (detects TanStack, NextJS, etc. from package.json) |
| **Exclude patterns** | Built-in (e.g., _test.go, vendor/) | Configurable via `gazelle_ts.json` excludePatterns |
| **Runtime deps for tests** | Not supported | Supported via `runtimeDeps.test` in gazelle_ts.json |

**Friction point:** Gazelle_ts.json introduces an extra configuration file that developers must remember exists. Go developers are accustomed to `gazelle:` directives only.

## SUMMARY TABLE: Quick Reference

| Workflow | rules_go | rules_typescript |
|----------|----------|-----------------|
| **1. Project bootstrap** | Simple; minimal .bazelrc | Complex; must add explicit return types |
| **2. Add dependency** | `go get` + `bazel run //:gazelle` | `pnpm add` + commit pnpm-lock.yaml + update MODULE.bazel |
| **3. Write library** | .go files; Gazelle generates BUILD | .ts files + index.ts barrel; explicit return types required |
| **4. Write test** | `*_test.go` files; built-in testing pkg | `.test.ts` files; requires @npm//:vitest in deps |
| **5. Cross-package imports** | Absolute paths (`my_project/...`) | Relative paths (`../...`); .d.ts compilation boundary |
| **6. Binary** | Static executable via go_binary | JS bundle via ts_binary; requires Node.js at runtime |
| **7. Type checking** | Automatic, mandatory | Optional; configured via .bazelrc `--output_groups=+_validation` |
| **8. IDE integration** | Automatic via gopls + go.mod | Manual via `bazel run //:refresh_tsconfig` |
| **9. CI setup** | `bazel test //...` | `bazel build //... --output_groups=+_validation && bazel test //...` |
| **10. Dev server** | N/A | Skeleton (ts_dev_server); not implemented |
| **11. CSS/assets** | N/A | css_module, asset_library rules; auto-generate .d.ts |
| **12. Bundling** | Static link (implicit) | ts_binary/ts_bundle; requires Vite or custom bundler |
| **13. Coverage** | Built-in via `bazel coverage //...` | Explicit @vitest/coverage-v8; separate coverage target required |
| **14. Gazelle directives** | `gazelle:prefix`, `gazelle:ignore`, etc. | `gazelle:ts_package_boundary`, `gazelle_ts.json` per-directory config |

## KEY ERGONOMIC GAPS IN rules_typescript (vs rules_go)

1. **Explicit type annotations** — Developers must annotate all exports with types (isolated declarations requirement). Go has implicit typing.

2. **Barrel file discipline** — Must maintain index.ts files as package boundaries. Go has implicit package-based boundaries.

3. **Gazelle refresh** — Must run `bazel run //:refresh_tsconfig` after adding targets for IDE support. Go's gopls does this automatically.

4. **Package manager required** — pnpm is mandatory. Go's go.mod is native.

5. **Test npm deps** — Must remember to list npm packages in ts_test deps. Go's imports are automatic.

6. **Coverage setup** — Requires separate test target + explicit npm dep. Go's coverage is automatic.

7. **No native binary** — ts_binary produces .js requiring Node.js, not a static executable. Go produces stand-alone binaries.

8. **Dev server not implemented** — ts_dev_server is a skeleton. No live-reload support.

9. **Configuration explosion** — More .bazelrc flags, gazelle_ts.json, tsconfig.json, vite.config.ts. Go is simpler.

10. **Import style switch** — Absolute imports (Go) vs. relative imports (TypeScript). Mental model differs.

---

## HOW rules_typescript ACHIEVES GO-LIKE ERGONOMICS

Despite the gaps, rules_typescript successfully mirrors Go's ergonomics in key areas:

1. **"Write .ts, let Gazelle generate BUILD"** — Parallels Go's `gazelle -r .`

2. **Type errors fail the build** — Both use `bazel build` for compile + type-check

3. **Fine-grained incremental caching** — Isolated declarations enable .d.ts boundary; similar to Go's package-based compilation

4. **Single command workflow** — `bazel build //...` builds everything, including types

5. **No npm/pnpm runs during build** — Fully hermetic; pnpm-lock.yaml is immutable
